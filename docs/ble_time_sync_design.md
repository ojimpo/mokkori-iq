# BLE 時刻同期 — 設計ドキュメント (Phase 2 / 最初の通し実装)

スマホ（iPhone / Swift ネイティブ）からデバイスへ「現在時刻」を渡し、フラッシュに
記録したサンプルへ**絶対時刻**を割り当てられるようにする。XIAO nRF52840 は電源 OFF 中に
時計を保持できない（RTC バックアップ電源が無い）ため、外部＝スマホから基準時刻をもらう。

これは将来構想「水上 BLE オフロード → スマホアプリ」の最初の一歩。まず「時刻合わせ」だけを
ファーム → ツール → アプリまで端から端まで通して動かし、BLE の土台を作る。データ吸い出しの
characteristic は後で足す。

## 方針の決定（このドキュメントで確定したこと）
- **プラットフォーム = iPhone / Swift ネイティブ（SwiftUI + CoreBluetooth）**。
  iPhone 専用で、取込用 Ubuntu タブレットも BLE 化で不要になるため、Flutter の
  クロスプラットフォーム利点はこの案件では効かない。アプリの核が自作 BLE 機器との
  通信なので、抽象層を挟まない CoreBluetooth が一番ハマりにくい。
- **時刻合わせの起動はタップ（叩く）で行う**。物理ボタンは防水ケースに穴が要る
  （浸水リスク）ので、今あるハードのまま＝加速度センサのタップ検出をトリガーにする。
  物理ボタンは筐体設計のときに改めて検討する。
- **BLE はタップで起動した時だけ開く**（常時 ON でも電源 ON 時だけでもない）。叩いて
  起こした時に数十秒だけ BLE を開く。サンプリングしていない時に操作するので、52Hz の
  記録と無線処理がぶつかる心配が無い。

## 前提・制約
- **時刻合わせはプールサイドの陸上で行う**。水中は 2.4GHz が水に吸収されるので BLE は
  水上専用（元々の方針）。デバイスを手に持った状態で叩いて起こし、スマホで合わせてから
  股間に仕込んで泳ぐ、という今の運用に乗る。
- 同期はセッションに1回でよい（最初に1回合わせれば全サンプルに時刻を振れる）。再同期も可。
- この段階では吸い出しは従来どおり USB のまま。**時刻アンカーだけ BLE で入れる**。
- 既存のフラッシュ形式（256B / magic `0xA55A` / 20 sample）と Phase 0 取込パイプラインは
  **壊さない**。時刻は別レコード・別コマンドで足す。

## 時刻モデル（アンカー方式）
デバイスは絶対時計を持てないので、`millis()` 起点の相対時間に**アンカー（基準点）**を
1つ以上打つ。

- アプリが epoch をデバイスに書き込む（時刻同期 characteristic）。
- デバイスは書き込み受信時点を `(epoch_ms = E, サンプル番号 = K, millis = m)` として
  フラッシュに**追記**する（専用 magic `0xA55B` ページ。下記）。
- ホストは各サンプル i の絶対時刻を、i 以前で最も近いアンカーを使って復元:

  ```
  unixtime_ms(i) = E + (i - K) * 1000 / ODR_HZ
  ```

- 再同期すれば複数アンカーが入る（時計ずれの補正に使える）。`millis` は将来の誤差推定用に
  残すだけで、最初の実装では K と E だけで足りる。

### なぜサンプル番号 K に紐付けるか
記録は追記式でサンプルは連番。アンカーは「**ここまでに K サンプル書いた時点が E だった**」
という事実だけ保存すれば、ホストが index→絶対時刻を一意に逆算できる。1サンプルあたり
`1000/ODR_HZ` ms 進む前提（ODR は INFO が報告）。

## タップ・トリガーの設計（誤作動させない）
今ある加速度センサだけで、「時刻合わせを始めたい」という意思表示を拾う。**既存の3タップ
境界マーカー（壁でのトントン）とも、泳ぎの加速度とも、ぶつからない**ことが要件。

- **静止ゲート**: 直前に数百 ms〜1s ほど `|acc|≈1g`・低ぶれの「ほぼ静止」を要求する。
  手に持ってデッキで操作する前提なので、泳ぎ中の加速度では成立しない。
- **専用パターン**: タップ数を境界マーカーの3とは**変える**（例: 意図しないと出ない長めの
  連打＝5タップ等）。等間隔の鋭い jerk スパイクの連続として検出。
- 検出したら BLE アドバタイズを数十秒開く（LED で「同期待ち」を表示）。スマホが繋いで
  時刻を書いたら閉じる。タイムアウトでも閉じる。
- パターン・しきい値は実機データで詰める（既存 `turn_stand.py` のタップ三連検出が出発点）。
  境界マーカーとの取り違えが無いことを実信号で確認する。

## フラッシュ形式の拡張（後方互換）
既存のサンプルページ（magic `0xA55A`）はそのまま。**新しいアンカーページ**を同じ追記
ストリームに混在させる:

```
[0:2]  magic 0xA55B            // anchor page（サンプルページの 0xA55A と区別）
[2:4]  count = 0               // 0 = サンプルを含まない（既存 count>0 と区別）
[4:6]  seq                     // 既存 seq と連番共有
[6:8]  reserved
[8:16] epoch_ms  uint64 LE     // アプリが書いた UTC epoch（ミリ秒）
[16:20] sample_index uint32 LE // このアンカー時点のサンプル番号 K
[20:24] millis_at_sync uint32  // 受信時の millis()（将来の誤差推定用）
[24:256] 0
```

- `scan_log()` の変更: magic `0xA55A` → 従来どおりサンプルページ（`count` 本加算）。
  `0xA55B` → アンカーページ（サンプル0本、ページは消費して継続）。それ以外 → ログ終端。
- 「ログの途中にアンカーが挟まる」追記が自然に表現でき、追記哲学とも整合。

## ホスト/ツール側の変更
- DUMP プロトコルは**変えない**（サンプル12Bのみを流す）。アンカーは別コマンドで取得:
  - 新コンソールコマンド `TIMES`: アンカーを行で出力（USB）。例:
    ```
    TIME epoch_ms=1718954029000 sample_index=12480 millis=240115
    OK
    ```
- `tools/flash_dump.py`: `--pull` 時に DUMP（サンプル）＋ TIMES（アンカー）を取得し、
  CSV に **絶対時刻列を追加**する（既存 `idx,t,ax..gz` に `unixtime` を足す）。アンカーが
  無いセッションは従来どおり。
- `dataio.load_swim_csv`: unixtime 列があれば読む（無くても従来動作）。swim_report の
  `--date` は将来この絶対時刻で自動化できる。

## BLE GATT 設計
カスタム 128bit UUID。base はプロジェクト固有値（`6D6B6F69` ≈ "mkoi" = mokkori-iq、
`A55A` はフラッシュ magic に由来）:

- **Service** `6D6B6F69-A55A-4000-8000-000000000001` … mokkori-iq logger

| Characteristic | UUID 末尾 | プロパティ | 内容 |
|---|---|---|---|
| 時刻同期 (set) | `...0002` | Write w/ response | 8B uint64 LE = UTC epoch_ms。受信でアンカー追記 |
| 状態 (get)     | `...0003` | Read / Notify | 現在推定時刻・最終アンカー・samples・vbat 等 |
| データ吸い出し | `...0004` | Notify | **将来**: DUMP 相当をストリーム |
| 制御           | `...0005` | Write | **将来**: ERASE / セッション境界マーク 等 |

最初の実装は **時刻同期(0002)** と **状態(0003)** のみ。0004/0005 は予約。

## アプリ（iPhone / Swift）側 最初のスコープ
- CoreBluetooth セントラル。Service UUID でスキャン→接続。
- 「時刻を合わせる」ボタン: 現在の `Date()` を epoch_ms にして 0002 へ Write、0003 を読んで確認表示。
- 状態表示: 接続状態・最終同期時刻・記録量(samples/秒)・電池(vbat)。
- まだやらない: データ吸い出し(0004)、ログ閲覧 UI、Strava 連携（後続）。

## 実装の順番
1. **本ドキュメント**（設計確定） ← いまここ
2. ファーム: BLE スタック導入（Bluefruit nRF52）+ Service/0002/0003、タップ検出→同期窓、
   アンカー追記、`scan_log` 対応、`TIMES` コマンド。記録と BLE が干渉しないこと・タップ誤作動が
   無いことを実機確認。
3. ツール: `flash_dump.py` を TIMES 対応＋CSV に unixtime 列。`dataio` 読み取り対応。
4. アプリ: Swift プロジェクト雛形 + CoreBluetooth で 0002 書き込み / 0003 確認。
5. 実機通し確認: アプリで同期 → 少し記録 → USB で pull → CSV に正しい絶対時刻。

## 未確定・要検証（実機）
- **BLE スタックと TinyUSB CDC の共存**: Seeed/Adafruit nRF52 コアは BLE+TinyUSB 同居が
  一般的だが、本ファーム（QSPI flash + LSM6DS3 + CDC）と一緒に動くか実機確認が要る。
- **タップ検出の誤作動**: 既存の3タップ境界マーカー・泳ぎの加速度と取り違えないこと、
  静止ゲート＋専用パターンで十分か、を実データで確認・調整。
- **電池消費**: 同期窓のアドバタイズは短時間なので軽いはずだが、実測で確認。
- **配布**: iOS 実機インストールは Xcode 署名が要る。無料 Apple ID は証明書 7日失効、
  常用するなら Apple Developer Program（年 $99）で失効回避。
- **タイムゾーン**: フラッシュには UTC epoch_ms を保存。表示はアプリ/ホストでローカル変換。
- **物理ボタン**: 筐体設計フェーズで、防水を保てる押しボタン（または膜越し）の採否を再検討。
