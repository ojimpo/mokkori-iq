# mokkori-iq Phase 0: 水泳ターン検知アルゴリズムのプロトタイプ開発

## プロジェクト概要

「mokkori-iq」は水着の中に装着するDIYウェアラブルスイムトラッカー。日本の市民プールではGarmin等のスマートウォッチを着けて泳ぐことが禁止されているため、水着に仕込めるデバイスでスイムログ（ラップ数・ラップタイム・距離）を記録し、Strava等にアップロードできるようにするのが最終目標。加えて、泳いでいる最中にERM振動モーターで距離通知（例: 100mごと）を行う。

- **MCU**: Seeed XIAO nRF52840 Sense（オンボードIMU: LSM6DS3TR-C, 6軸）
- **装着位置**: 股間（仙骨に近いポジション）
- **最終的なデータフロー**: IMUでターン検知 → ラップタイムをフラッシュに記録 → USB-C or BLEでPCに転送 → .fitファイル生成 → Strava APIアップロード
- **リアルタイム機能**: 4ラップ(100m)ごとにERM振動モーターで触覚通知
- **泳法**: 自由形（クロール）のみ。他の泳法への対応は不要
- **ターン方式**: タッチターン（壁に足をついて反転して蹴り出す）。フリップターン（クイックターン/前転）はしない

## このタスクのゴール（Phase 0）

プールに行く前に、公開データセットを使ってPythonでターン検知アルゴリズムを開発・検証する。最終的にArduino/C（Cortex-M4, 256KB RAM）に移植するので、**閾値ベースのヒューリスティック**をまず作る。TFLite/機械学習は精度不足の場合の後段オプションであり、最初のアプローチではない。

## Step 1: データセットの取得と理解

### 主データセット: Brunner et al. (ISWC 2019)
- **リポジトリ**: https://github.com/brunnergino/swimming-recognition-lap-counting
- 40人分のスマートウォッチIMUデータ（手首装着）
- センサー: 加速度・ジャイロ・磁気・気圧・照度
- **30Hzにリサンプル済み**
- ラベル: 0=null, 1=freestyle, 2=breaststroke, 3=backstroke, 4=butterfly, **5=turn**
- リポジトリをcloneし、データ構造（ファイル形式、カラム配置、被験者数）を確認してから先に進むこと

### 参考論文: Delhaye et al. (2022)
- 論文: https://www.mdpi.com/1424-8220/22/15/5786
- **仙骨（sacrum）装着の単一IMU** → mokkori-iqの装着位置に最も近い先行研究
- 前処理: 2次バターワースローパスフィルタ（10Hz）→ 50Hzダウンサンプル
- 8クラス分類でF1=0.96達成
- この論文の前処理パイプラインを参考にすること（ただしモデル自体は不要、あくまで前処理の参考）

## Step 2: データ探索と可視化

Brunnerデータセットを使って以下を分析する。ユーザーは自由形しか泳がないため、**自由形（label=1）区間のターン（label=5）に焦点を当てる**。ただし他の泳法のターンとの信号差を比較することで、自由形フリップターン特有のパターンを浮き彫りにする分析も有効。

1. **ターン（label=5）周辺の加速度・ジャイロ波形を可視化**
   - ターン前後±5秒程度の時系列プロット
   - まず自由形区間のターンで共通するパターンを特定、次に他泳法のターンと比較
   - 加速度ノルム（√(ax²+ay²+az²)）の挙動に特に注目
   - ユーザーのターン方式はタッチターン（壁に足ついて反転）であり、フリップターン（前転）ではない。タッチターンの信号パターンは「ストローク減速 → 壁タッチで一瞬の低活動区間 → 壁蹴りの加速ピーク」になるはず。この減速→静止→加速のシーケンスを主要な検出特徴として探ること

2. **ターンの信号的特徴を定量的に把握**
   - 壁キック時の加速度ピーク（スパイク）の振幅・持続時間
   - ターン動作中のジャイロ変化（ピッチ反転等）
   - ターン前後の「静止区間」（壁タッチ→キック）の有無と長さ
   - 通常ストロークとターンの信号がどう異なるか

3. **被験者間のばらつき評価**
   - 自由形ターン信号の振幅範囲、タイミングのばらつき
   - Brunnerデータセット内でフリップターン vs タッチターンの信号差を確認（被験者によってターン方式が異なる可能性がある）。ユーザーはタッチターンなので、タッチターンのパターンがアルゴリズムのターゲット
   - タッチターンの「低活動区間」の長さのばらつきは検出ウィンドウ設計に直結するので定量化する

探索結果は `analysis/` ディレクトリにmatplotlibでプロットを保存し、発見した特徴パターンの要約をmarkdownで残すこと。

## Step 3: ターン検知アルゴリズムの実装

### 方針: 閾値ベースのヒューリスティック

Step 2の探索結果に基づいて、以下のようなアプローチでターン検出器を実装する（探索結果次第で具体的手法は調整してよい）:

- **候補アプローチ例**:
  - 加速度ノルムのピーク検出（scipy.signal.find_peaks等）
  - スライディングウィンドウ内の分散/エネルギー変化
  - ジャイロの急激な変化との組み合わせ
  - Delhaye論文を参考にしたバターワースフィルタによる前処理

- **アルゴリズムの出力（ラップログ構造を意識）**:
  - ターン検出は最終的にスイムログ記録の手段。検出器はターンのタイムスタンプのリストを出力すること
  - ターンのタイムスタンプ列からラップタイム（ターン間の経過時間）を算出するユーティリティも作る
  - 最終的にデバイス上では `{lap_number, lap_time_ms, timestamp}` のような構造体をフラッシュに書き込む想定
  - 評価時もこのラップログ形式で結果を出力し、「ラップ数が正しいか」「ラップタイムの誤差はどの程度か」を確認する

- **制約（Arduino移植を前提）**:
  - numpy/scipy等のライブラリ依存をアルゴリズムのコア部分では最小化
  - コアロジックはfor文+基本演算で書けるレベルにする（フィルタ係数は事前計算でOK）
  - 「因果的」（causal）な処理にする。未来のデータを参照しない。リアルタイムで逐次処理できること
  - メモリ使用量を意識: 数秒分のリングバッファで処理が完結するように

### 実装構成

```
mokkori-iq/
├── data/               # Brunnerデータセット（clone先 or シンボリックリンク）
├── analysis/           # 探索的分析のプロット・ノート

│   ├── lap_logger.py   # ターンタイムスタンプ列 → ラップログ変換
│   └── evaluate.py     # データセット全体での評価パイプライン（ターン検出+ラップログ両方）
├── notebooks/          # Jupyter notebook（探索用、オプション）
└── README.md           # 手法の説明、評価結果のサマリ
```

## Step 4: 評価

全40被験者の**自由形区間**に対して以下を評価（他の泳法は参考データとして二次的に評価してもよい）:

### ターン検出レベル
- **検出率（Recall）**: 実際のターンのうち何%を検出できたか
- **誤検出率（Precision）**: 検出したもののうち何%が本物のターンだったか
- **F1スコア**
- **検出タイミングの誤差**: ラベルとの時間差（秒）

### ラップログレベル（実用上こちらが重要）
- **ラップ数の正誤**: 各被験者のセッションで、ターン数（＝ラップ数-1）が正解と一致するか
- **ラップタイムの誤差**: 正解ラベルから算出したラップタイムとの差（秒）
- **累積誤差**: セッション全体を通して誤差が蓄積していかないか

- 被験者別のブレークダウン

目標: ヒューリスティックでターン検出F1 ≥ 0.85、かつラップ数の一致率 ≥ 90%を目指す。装着位置が異なる（手首 vs 股間）ため、このデータセットでの精度に過度にこだわるより、「自分のデータ（Phase 1）に持っていける汎用的なアルゴリズム構造」を優先する。

## 重要な注意点

- **自由形タッチターンに最適化する**: ユーザーのターンは「壁に足ついて反転して蹴り出す」タッチターン。フリップターン（前転）のような大きなジャイロ回転は発生しないため、加速度ベースの「減速→静止→加速」パターン検出が主軸になる。ジャイロは補助的な特徴として検討する。Brunnerデータセット内にフリップターンとタッチターンの両方が混在している可能性があるので、まず両者の信号差を把握した上で、タッチターンのパターンに寄せたチューニングを行うこと
- **装着位置の違いを常に意識**: Brunnerデータは手首装着。mokkori-iqは股間装着。ターン時の加速度パターンは異なるはずなので、「手首データで完璧に動く」ことより「ターンの一般的な信号特徴（急減速→回転→急加速）を捉える構造」を重視する
- **パラメータは設定ファイルに外出し**: 閾値・ウィンドウサイズ等はハードコードせず、Phase 1で自分のデータに合わせて調整しやすくする
- **MCUの制約を忘れない**: 最終ターゲットはCortex-M4 (64MHz, 256KB RAM)。浮動小数点演算は可能だがFPUはある。FFTや行列演算のような重い処理は避ける
