# 壁検出アルゴリズム 試行ログ（採用案・不採用案と経緯）— 2026-06-21

初プール実データ `swim_20260621_171349.csv`（股間装着・46.3分・52Hz）で、ターン/距離/ペース/SPM を
出すまでに**試した案を時系列で全部**記録する。最終採用は gz オリエンテーション壁検出（`turn_stand.py`）＋
保守ペース推定（`swim_report.py`）。不採用案も「なぜダメだったか」を残す（将来の再検討・別装着位置で復活し得るため）。

前提として、このセッションは**外部グラウンドトゥルースが無い**（プールにスマホ持込不可・本人も覚えていられない＝
だからこの装置を作っている）。3タップが唯一のGT機構だったが水中で失敗（後述#3）。よって全案は
「内的整合性（ターン等間隔）＋物理的妥当性＋妥当ペース」でしか評価できなかった。

---

## 採用までの試行（時系列）

### #1 既存の手首/30Hz検出器をそのまま適用 — 不採用（想定通り）
- 何を: Phase 0 の `detector.py`（Brunner手首データでチューニングした |acc|/activity の causal FSM）を
  `explore_swim.py` 経由で実データに適用。
- 結果: **検出 0 件**。|acc|平均は ≒1g で信号自体は健全。
- なぜ不採用: 手首・30Hz 前提のチューニング。股間・52Hz は信号が別物。Phase 1 計画でも「いきなり移植せず
  実データ採取が先」としていた通り。→ 別アプローチが必要と確定。

### #2 アクティビティ包絡のヒステリシス分割（`segment_swim.py`） — 部分採用（粗構造のみ）
- 何を: |acc| の trailing rolling std を活動量とし、パーセンタイル・ヒステリシスで「泳ぎ/休憩」を二値化。
- 結果: **2つの大ブロック＋約9.5分の大休憩**という粗構造は綺麗に取れた。が、個々の100mレップは
  **マージされてしまう**（レップ間の休憩が短すぎ＝タップして泳ぎ続けたため、閾値を割らない）。
- 判断: 粗構造の把握には有用なのでツールは残置。ただし**ラップ計数には使えない**（泳ぎ/休憩ブロックしか出ない）。

### #3 jerk インパルスによる3タップ検出（`segment_swim.py` 内 + 探索） — 不採用（重要な失敗）
- 何を: |acc| の jerk（時間差分）スパイクを低活動窓で拾い、約1秒間隔の3連クラスタ＝タップ境界とみなす。
- 結果: **冒頭の陸上タップ（10–17s付近）しか復元できない**。水中タップは完全に埋もれる。
  大休憩境界（1516–1522s）を拡大しても、壁タッチ/ストロークの大スパイクと混在して「等間隔3連」を分離できない
  （泳ぎ中の jerk は常時99パーセンタイル閾値を超える）。
- なぜ不採用: **境界マーカー機構として破綻**。これが「唯一のGT」だったので痛い。
- 後日の知見: タップは**立位で打てば** gz 領域で復元できた（#8参照）。jerk領域がダメなだけだった。
  → 採取手順を「水中で打つ」から「壁で足をついて立位で打つ」へ改訂（CLAUDE.md 反映済み）。

### #4 アクティビティ・ディップ＝壁、という素朴なターン検出 — 不採用
- 何を: 活動量包絡の局所最小（ディップ）を壁とみなす。8秒不応期つき。
- 結果: **136ターン**検出、間隔 約8–15s。それを25mとすると **8–15s/25m＝世界記録級ペース**で物理的にあり得ない。
- なぜ不採用: 壁以外（ストローク/呼吸の周期的ディップ）を大量に拾う**過大カウント**。閾値依存で脆い。

### #5 自己相関で「1本の周期」を客観推定 — 不採用（手がかりにはなった）
- 何を: 泳ぎ区間の活動量包絡を自己相関し、25mレングスに相当する支配周期を探す。
- 結果: ACFピークが探索下限（8s）に張り付き、相関も弱い（0.2台）。**25mスケールの明確な周期は立たない**。
  一方で短ラグに ストローク周期 約2.1s（28–29/分）が明瞭。
- なぜ不採用: 活動量包絡には使えるレングス周期が無い（ストロークリズムが支配的）。
  → ただし「ストローク周期は綺麗に出る」と分かり、後の **SPM計測（#15）の伏線**になった。

### #6 フリップターンの回転検出 — 不採用（泳法の聞き取りに直結）
- 何を: ジャイロを2秒窓で積分し、|正味回転| > 2.5 rad（≒宙返り）のイベント＝ターンとみなす。
- 結果: 全セッションで**わずか11イベント**。ターン毎には程遠い。
- なぜ不採用: 採取者は**オープンターン（クイックターン不可）**で宙返りしない。回転シグネチャが存在しない。
  → これがきっかけで泳法を確認 →「毎回足をついて立つ」「100m境界は水中で数秒静止+トントン／ベンチ着座はハーフ間の長休憩のみ」が判明 → #8 の決定打に。

### #7 グライドギャップ検出（`turn_glide.py`、削除済み） — 不採用（最有力の次点だった）
- 何を: ターン＝プッシュオフ後の**ストリームライン滑走**。短窓(0.6s)のストローク活動が局所ベースライン(6s)の
  40%未満に落ち、両側を泳ぎに挟まれた区間＝壁。不応期14s。
- 結果: ギャップ間隔が **約30–40sにクラスタ**（＝妥当な25mペース）で、初めて「物理的にもっともらしい」値が出た。
  ただし偽陽性（サブレングスの静止＝呼吸/グライド）と取りこぼしが残る。
- なぜ不採用: #8 の gz オリエンテーションの方が**遥かに明確で物理的に堅い**（オープンターンの足つき立位は
  グライドより圧倒的に強い・曖昧さの少ない信号）。`turn_glide.py` は採用後に削除し、本ログに思想を保存。
  別装着位置（手首/ゴーグル等、立位で体軸が垂直化しない位置）では**グライドギャップが復活し得る**ので、
  手法として記録しておく価値がある。

### #8 gz 体軸オリエンテーション壁検出（`turn_stand.py`） — ★採用
- 何を: 立位/着座で股間デバイスの体軸が垂直化 → **重力のz成分が 泳ぎ -7.5 ↔ 立位 +9.5 m/s² に振れる**。
  この gz スイングで壁を取る。gz>2 の立位エピソードを分類。
- なぜ採用: グライド/回転/jerk のどれより明確。`turn_stand.py` 内のコメント・`findings_2026-06-21` 参照。
  3タップも**立位で打てば gz の三連ブリップとして分離できた**（901s/938s）＝#3の失敗を回収。
- 装着向き依存の注意: USB-C上/Seeed面外側 前提。向き未知なら「泳ぎ⇄静止で最も振れる軸の自動選択」へ一般化が要る。

## 採用案の中での調整（試して直したこと）

### #9 gz分類の初版（閾値交差のみ）→ 過大カウント。ピークgz>5＋不応期12s を追加
- 症状: 2520–2620s に **約2秒間隔の立位ブリップ約20連発**（gzが閾値付近を小刻みに往復）を全部ターンに数え、
  block2 が ~1500m に膨張。
- 原因: 半立ち/水中ボビングで gz が中途半端に正になる区間。
- 対策: **ピーク gz > 5（明確に直立）**を要求して半立ちを除外＋**不応期12s**（1レングス最短時間の床）。
  → block2 が 825m まで収束。

### #10–#13 距離の数え方（過大↔過小のトレードオフ）
| 案 | 距離 | 評価 |
|---|---|---|
| #10 壁カウント生 | 1350m | 偽分割（壁の二重カウント）で**過大**。不採用 |
| #11 ペースベース round（レップ泳ぎ時間/標準レングス時間） | 1125m | 偽分割は消えるが #14 のバグあり |
| #12 floor 一律（保守） | 950m | **過小すぎ**。検証済みの綺麗な100mまで75mに削る。不採用 |
| #13 **min(壁カウント, round(ペース))** レップごと | 1150m | ★採用。下記 |
- 採用理由（#13）: 偽分割の過大は**ペース項が頭打ち**にする。壁取りこぼしは壁カウントが低く出て **min がそれを採用＝過小（許容方向）**。
  綺麗なレップは両項一致で**正確に保持**（floorのように削らない）。距離は下振れ・ペースは遅め＝[メトリクス算出方針]に合致。

### #14 「標準レングス時間」の単位ミスマッチ（バグ→修正）
- 症状: 初版の標準レングス時間を**ターン開始間隔**で測ったら 35.1s になり、検証済みの綺麗な100mまで 75m に削れた。
- 原因: 開始間隔には**立位時間(約3s)が混入**。一方 分子は「泳ぎ実時間(立位除外)」。単位不一致で分母が過大→本数過小。
- 修正: 標準レングス時間＝**ターン間の泳ぎ実時間（push-off .end → 次 .start の prone 時間）の中央値 = 28.5s/25m**。
  分子と単位を揃えたら綺麗な100mが正しく4本に戻った。

### #15 SPM（ストローク率）— norm は不採用、符号付き単軸を採用
- 何を: #5 で見えたストローク周期を計測。泳ぎ区間でジャイロのロール軸のスペクトルピーク＝ストロークサイクル率。
- norm を不採用: **|acc|/|gyro| は整流で 2倍/4倍の harmonics** が出る（|acc|が112/min, |gyro|が57/min 等）。
  → 符号付き単軸（ロール軸）で取り、基本周期 0.479Hz=**30 cyc/min（≒60 strokes/min）**を確定。3手法一致（窓FFT/自己相関/トラッカー）。
- 既知の限界: 8秒窓のFFT分解能 0.125Hz=**7.5 cyc/min 刻みに量子化**。精度が要るなら自己相関/放物線補間/長窓。
