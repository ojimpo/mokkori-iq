# 検討: 泳ぎながら100m毎にリアルタイム振動フィードバック — 2026-06-21

振動モーター未着のため**実装前の実現可能性検討**。因果オンライン版を録画データで模擬した
（`analysis/realtime_sim.py`、対象 `swim_20260621_171349.csv`）。**結論: 実現可能・筋が良い。
ただしオープンループのカウントドリフトが唯一の実課題**で、対策候補も明確。

## なぜ筋が良いか（リアルタイム適性）
1. **適応閾値が要らない**: 採用した gz 壁検出は**絶対重力しきい値**で動く（直立=gz>+5、伏せ泳ぎ=gz<-2）。
   活動量ベース検出器のような「全セッション統計での較正」が不要 → **1サンプル目から因果実行可能**。
2. **MCU 軽量**: gz は acc_z の causal EMA（O(1)・先読みなし）。判定は約20行の状態機械。浮動小数の sqrt も不要。
3. **発火タイミングが理想的**: 壁で**直立を確認した瞬間**に発火＝スイマーが足をついて止まっている時に振動が届く。
   「今100m来た」を壁で受け取れる → **休むか続けるかの判断に間に合う**（オフラインのラップログより価値が高い使い方）。
4. **オープンターン前提と相性が良い**: 毎回足をついて立つ＝各壁が明確な立位イベント → 壁の取りこぼしが起きにくい。

## 模擬結果（このセッション）
- causal 壁カウント **50本（1250m相当）**、100m毎の振動 **12回**、間隔 135–175秒（≒妥当な100m時間）。
- 大休憩中は鳴らない（760秒ギャップ＝正しい挙動）。状態機械が休憩を自然に無視。

## 唯一の実課題: オープンループのドリフト
先読みが無いので、オフラインの「ペース/min() 補正」が使えない。**1本の誤検出/取りこぼしが以降の発火を恒久的にずらす**。
- 本セッション: causal 1250m vs オフライン保守 1150m ＝ **約+100m(+8%) 過大**。
  主因: ①ウォームアップ区間の壁も数える ②たまの水中ボビング誤検出 ③連続泳ぎ区間（足つき無し）。
- 影響度: 「ざっくり100m毎の体感フィードバック」用途なら ±25m 程度のズレは許容範囲かもしれない（要本人評価）。
  ただしセッションが長いほどドリフトは累積する。

## ドリフト対策の候補（実装時に段階導入）
1. **スタートアンカー**: 泳ぎ始め（最初の安定した伏せ泳ぎ）までカウント開始を遅らせ、ウォームアップの誤カウントを除外。
2. **再同期アンカー**: 100m境界は「長め立位 7–17秒＋トントン」で因果的にも識別可能（[境界2信号]の知見）。
   トントン or 長立位を検出したら、直近の壁カウントを 4 の倍数へスナップしてドリフトを定期補正。
   ＝**トントンがリアルタイム再同期の鍵**として復活する（オフラインでは任意だったが、ここで価値が出る）。
3. **しきい値の追い込み**: GZ_VERT/不応期を上げて水中ボビング誤検出を削る。
4. **連続泳ぎの扱い**: 足をつかず泳ぎ続けると壁を落とす。運用で「各壁で一瞬足をつく」を維持すれば回避（本人は既にそうしている）。

## 検証の不足（実装前にやること）
- **1セッションのみ**での評価。複数セッション・別日・別プールで再現性とドリフト量を測る必要がある。
- 振動の知覚: 股間装着の振動が泳ぎ中に確実に分かるか（水の抵抗・集中下で）。モーター強度・パターン（例: 100m=1回, 500m=2回）の検討。
- 誤発火コスト: 余計に1回鳴る vs 鳴らない、どちらが体験上痛いか。

## 実装ロードマップ（モーター着後）
1. まず `realtime_sim.py` のロジックを C に移植（gz EMA + 状態機械 + 不応期）。距離は壁カウント*プール長。
2. **最小版**: 壁を数えて4本毎に振動（対策#1スタートアンカーのみ）。実泳で体感評価。
3. ドリフトが気になれば **#2 トントン再同期**を追加。
4. しきい値・振動パターンを実泳でチューニング。
